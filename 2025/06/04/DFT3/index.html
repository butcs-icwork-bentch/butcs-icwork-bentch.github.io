    • bist_done：测试完成标志
    • bist_pass：测试通过标志

LBIST的局限性

这个实现是不完备的，主要局限在于：
缺少扫描链：完整的LBIST应包含扫描链，以提高可测性，这个设计直接将八位伪随机数的前四位给a后四位给b这样进行输入，输出则是通过misr压缩之后和标志位进行比对。可以看到，输入是电路在func状态下的输入端口，而不是在test状态下的扫描链结构。DFT2中我们说过扫描链模式mux选择是连接scan_en的，而bist模式的打开只需要test_mode 和bist_en就可以打开，此时scan_en是为0的，也就是打开了func模式，因此，从这一角度也能说明不是一个完备的LBIST设计。那么怎么样才能算是一个完备的LBIST呢？
我们看看DFT COMPILER USER GUIDE 是怎么说的。

可以看到
正确的顺序应该是先产生伪随机码通过scanchain输入芯片，scanchain输出到misr压缩向量成为标志位从而进行比对。原来如此，那么我们看到差异其实就是电路工作模式的问题，只需要改变电路从func到scantest即可，但是这里边就有了新的问题。
什么问题呢？
按道理来说EDA应该是不光能自己产生scanchain也应该能产生性能更加完美的BIST结构，但是对于一些库不太完备的pdk，或者一些小工艺厂，是没有sldb文件去做优化的，因此我们就需要从rtl或者网表级去加入BIST结构，主要包括，lfsr，misr，control center，三个部分，其实有点像UVM测试板块吧，只不过UVM板块不需要做出来，不需要misr压缩，其他都大差不差的。

可以看到这些stuck faults的类型及其含义如下：

  • Detected (DT)：测试向量能覆盖的故障。

至此fault的检测以及pattern生成结果我们大致已经知道了，在导出测试文件之前，我们有一个疑问，那就是为什么上边faults检测到了1014个，下边报告给出了1278个？

为什么上面检测到的faults是1014个，而下方报告显示1278个？

主要原因如下：

  • 统计口径不同：

    • 上面”patterns stored”那一行的1014 faults，通常指的是本次ATPG生成的测试向量实际检测到的故障数（即pattern级别的统计）。
    • 下面”Uncollapsed Stuck Fault Summary Report”中的1278 Detected，指的是所有理论上可以被检测到的故障总数（summary级别统计，可能包含等价故障、未折叠故障等）。

  • 故障模型的展开与未展开：

    • 有的ATPG工具会在summary报告中统计”未折叠（uncollapsed）”的所有故障（比如每个节点的SA0/SA1都算），而主表格可能只统计”折叠后（collapsed）”的故障数。

  • 等价故障与分组：

    • 某些故障在ATPG过程中被合并或分组（如等价故障），导致主表格和summary统计方式不同。

  • 测试模式和覆盖率算法不同：

    • 主表格只统计”当前pattern检测到的故障”，而summary会统计”理论上所有pattern能检测到的故障”。

总结：

  • 1014：实际被本次pattern检测到的故障数（pattern级别统计）。
  • 1278：所有理论上可检测的故障数（summary级别统计，可能包含等价故障、未折叠故障等）。

这两者的差异，主要是统计口径和故障模型的不同导致的。实际芯片测试时，通常以summary中的覆盖率为准。

STIL文件格式解析

TetraMAX生成的STIL（Standard Test Interface Language）文件是一种标准化的测试向量描述格式，用于描述设计的测试模式。下面详细分析STIL文件的主要组成部分：

1. 文件头部 (Header)

  • STIL版本声明及设计信息
  • 标题、日期和来源信息
  • 包含故障覆盖率摘要和测试统计数据的历史记录

2. 信号定义 (Signals)

  • 定义所有输入输出信号及其方向
  • 特殊属性标记（如ScanIn、ScanOut）
  • 包括控制信号、数据信号和扫描信号

3. 信号组 (SignalGroups)

  • 将相关信号组合在一起便于引用
  • 典型组包括：主输入(_pi)、主输出(_po)、扫描输入(_si)、扫描输出(_so)等

4. 时序定义 (Timing)

  • 定义信号时序和波形变化
  • 周期设置（本例为100ns）
  • 各信号在周期内的变化模式
  • 多个波形表用于不同测试阶段

5. 扫描链结构 (ScanStructures)

  • 定义设计中的扫描链
  • 包含长度、输入/输出端口、控制时钟等信息
  • ScanInversion参数指定是否进行数据反转（0表示不反转）
  • ScanCells列出链中所有触发器

6. 程序和宏 (Procedures/MacroDefs)

  • 定义测试步骤和操作
  • 包括捕获操作、扫描链加载/卸载等程序
  • 测试设置和初始化的宏定义

7. 测试模式 (Pattern)

  • 定义具体测试向量序列
  • 每个模式包含加载值和期望输出
  • 调用预定义的程序和宏
  • 注释说明测试目的

可以看到，这个STIL文件为我们的4位乘法器定义了27个测试模式，总共生成477个测试周期，覆盖率达到98.46%。STIL格式作为一种工业标准，可以用于多种自动测试设备(ATE)，实现测试向量的便捷共享和应用。
但是我们还没有达到生产阶段，肯定是先不能加载ate测试，于是，我们现在需要将stil文件转化成为我们可以仿真的文件，也就是testbentch格式的.v文件，这样就可以去做波形的测试，看是否输入和输出扫描链结果符合预期，只有仿真无误才能说明，atpg输出的测试向量是正确的，从而才能保证加载进ate可以得到正确的结果从而提高可靠性。
